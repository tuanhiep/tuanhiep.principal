# 🧱 Engineering Brick: Eradicating the "God Class" via Collection Injection

> 🌸 *Code flows long like an endless stream,*
> *One class bears the weight of a thousand tasks.*
> *Break the if-else loop, split into a hundred paths,*
> *Spring Boot gathers them, bringing peace.*

## 🌠 1. The Formal Specification (Problem Model)

Before discussing Design Patterns, we must confront the "disease" of Enterprise systems. Assume we are building a **Payment Routing Engine** for a multinational E-commerce platform.

**The Interface (Core Operations)**:
* `processPayment(PaymentContext ctx)`: Accept the request payload and route it to the correct payment gateway.

**The Workload & Constraints (Real-world Scale)**:
* **Business Scale**: The system supports 15 payment methods (Credit Card, PayPal, Apple Pay, Crypto, Buy-Now-Pay-Later...). The Business team demands a new integration every month.
* **Team Scale**: 5 different autonomous squads operate on this exact payment flow.
* **The Anti-Pattern**: The `PaymentService.java` file is currently 3,000 lines long, dominated by a monolithic `switch-case/if-else` block.

The Challenge: Design a **Plug-and-play** architecture that strictly adheres to the Open-Closed Principle (OCP). Adding a new payment gateway must require **Zero-Touch** on the core routing logic to completely eliminate merge conflict risks.

---

## ⚡ 2. The Design Dialogue (Socratic Review)

*In this section, we simulate a System Design Review between a Senior Engineer (The Challenger) and the Staff Architect.*

> **🕵️ The Challenger**: To resolve the if-else chain, why not use a `PaymentFactory` combined with an `Enum`? We create a `Factory.getStrategy(paymentType)` method that returns the correct implementation. It is simple and universally understood.

**🧑‍💻 The Architect**:
The Factory Pattern succeeds in splitting the file, but it merely shifts the bottleneck from the Service to the Factory. Every time a new payment method is added, 5 teams must still fight to modify the `PaymentFactory` file to register their new Enum. The code still violates OCP.

> **🕵️ The Challenger**: What if we use Reflection to automatically scan for classes implementing the interface at startup, and push them into an in-memory `HashMap`? It would be fully automated!

**🧑‍💻 The Architect**:
Writing custom Reflection logic is resource-heavy, prone to obscure bugs, and increases application startup time. We are standing on the shoulders of a giant—Spring Boot. Why reinvent the wheel when we can leverage its core superpower: the **IoC Container**?

---

## 🧩 3. The Architecture & Inversion of Control

Instead of manually instantiating and managing Strategies, we hand over complete authority (Inversion of Control) to Spring Boot via **Collection Injection**.

```java
// 1. The Contract (Invariant)
public interface PaymentStrategy {
    // The Sensor: Does this strategy apply to the current context?
    boolean supports(PaymentContext ctx);

    // The Execution: Process the business logic
    PaymentResult execute(PaymentContext ctx);
}

// 2. The Plugins (Distributed across different domain packages)
@Component
public class CryptoPaymentStrategy implements PaymentStrategy {
    @Override
    public boolean supports(PaymentContext ctx) {
        return ctx.getType() == PaymentType.CRYPTO;
    }

    @Override
    public PaymentResult execute(PaymentContext ctx) {
        // ... cryptocurrency transaction execution logic
        return PaymentResult.success();
    }
}
// O(1) time | O(1) space for internal plugin routing
